# Dotline — итоговый отчёт

## Что сделано

Выполнен полный цикл доработки Reverse Timing Dotline:

1. **Модальный выбор времени/даты** заменил `prompt()`.
2. **Модальный выбор интервала** между точками заменил inline-редактирование.
3. Время в Reverse Dotline переведено на **timestamp-модель**, добавлена миграция старых данных.
4. Точки увеличены, часы внутри них переписаны на надёжные `div`-стрелки.
5. Проведены четыре круга ревью: мобильная/десктопная версии, UI/UX, edge cases, продукт в целом.
6. Обновлена документация (`README.md`, `DEPLOYMENT.md`, `TIME_PICKER.md`).
7. Создан публичный репозиторий на GitHub: https://github.com/SmartShelfAI/dotline
8. Задеплоено на https://fungeneering.com/notes/

## Плюсы

- **Простота**: один HTML-файл, нет сборки, легко деплоить.
- **Мобильный UX**: свайп-удаление, тач-дружелюбные элементы, видимые кнопки удаления на карточках.
- **Гибкость Reverse Dotline**: anchor = текущее время, интервалы и абсолютное время редактируются отдельными picker'ами, поддержка других дней.
- **Синхронизация**: Google Sign-In + Flask backend + SQLite, last-write-wins.
- **Документация**: структура, деплой, логика pickers задокументированы.
- **Надёжность часов**: div-based стрелки не страдают от SVG transform-origin в разных браузерах.
- **Date picker**: нативный календарь открывается рядом со стилизованной кнопкой, а не за экраном.

## Минусы

- **Синхронизация last-write-wins**: одновременное редактирование на двух устройствах приведёт к потере одной из версий.
- **Нет офлайн-индикатора**: если backend недоступен, приложение просто работает локально, не предупреждая о несинхронизированных изменениях.
- **Нет undo/redo**: удаление точки или изменение времени нельзя отменить одной кнопкой.
- **Зависимость от нативных input'ов**: `<input type="time">` и `<input type="date">` выглядят по-разному в iOS/Android/Desktop; стилизация ограничена.
- **Секреты вне репы**: `GOOGLE_CLIENT_ID` хранится только на сервере, что правильно, но это означает, что локальный backend без настройки не заработает.
- **Большой `index.html`**: ~60 КБ в одном файле — приемлемо, но масштабировать сложнее.

## Риски

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| Потеря данных при last-write-wins | Средняя | Высокое | Добавить conflict-resolution UI или timestamps на уровне заметок/точек |
| Нативный date/time picker плохо работает в каком-то браузере | Низкая | Среднее | Тестировать на iOS Safari, Android Chrome, Firefox; иметь fallback |
| SVG/clip/scale issues на нестандартных экранах | Низкая | Низкое | Использовать div-based элементы там, где возможно (уже сделано) |
| GitHub-репо содержит чувствительные данные | Низкая | Высокое | `.gitignore` исключает `.env`, `me.jpg` и бэкапы; проверено перед пушем |
| Будущие изменения prototype/index рассинхронизируются | Средняя | Среднее | Использовать build-скрипт (`build.sh`) вместо ручной сборки |
| Долгосрочная поддержка SQLite на сервере без бэкапов | Средняя | Высокое | Настроить cron-дамп `sqlite3 dotline.db .dump` |

## Рекомендации

1. Добавить `build.sh`, который автоматически собирает `index.html` из `_head` + `prototype.html` + `_tail`, чтобы не делать это вручную.
2. Ввести версионирование данных (`dotline.v2`) и хранить `updatedAt` на уровне каждой заметки для более умной синхронизации.
3. Добавить undo для удаления точки (например, тост с кнопкой "Undo" на 3 секунды).
4. Настроить бэкап SQLite на сервере.
5. Добавить PWA-манифест и service worker для офлайн-работы.
